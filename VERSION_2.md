## ⚠️ CRITICAL: How This AI Agent Must Work

Read this section before doing anything else. These rules are mandatory.

### Rule 1 — Never write all 200 levels at once

Do NOT read all 200 levels and generate all content in a single pass.
Do NOT write a loop that processes all modules in one response.
Do NOT generate batch output and paste 200 debriefs into files.

**Why:** Content generated in bulk becomes templated and identical. Each level must be treated individually.

### Rule 2 — Use subagents for content generation

When generating content (Phase 2), spawn a separate subagent for each module.
Each subagent handles one module (15-20 levels) independently.
Wait for a subagent to finish and report quality before launching the next.

```
Main agent:
  Phase 1 → implement engine changes directly (code)
  Phase 2 → spawn Agent for module-1, wait, review output
           → spawn Agent for module-2, wait, review output
           → ...continue one module at a time
```

Subagent prompt template (fill in module name and path):

```
You are generating V2 content for K8sMissions module: [MODULE_NAME]
Path: modules/[MODULE_NAME]/

For each level in this module, in order:
1. Read broken.yaml, solution.yaml, mission.yaml
2. Read the existing hint files and debrief.md
3. Generate NEW content following the standards in VERSION_2.md Section 4
4. Write the files: debrief.md, hint-1.txt, hint-2.txt, hint-3.txt, investigation.yaml
5. Update mission.yaml with new_concepts if applicable (check which concepts were
   already introduced in earlier levels — the list is passed to you below)

Already-introduced concepts (do NOT add new_concepts for these):
[LIST FROM PREVIOUS MODULES]

Process one level at a time. Read, generate, write, then move to the next.
Do not summarize or batch. Write each file as you go.
Report back: list of levels processed and any new concepts introduced.
```

### Rule 3 — One level at a time within each subagent

Each subagent processes its module level by level:
- Read level files
- Generate content
- Write files
- Move to next level

Never read all levels first and then write all output.

### Rule 4 — Test engine changes before generating content

Phase 1 (engine code changes) must be complete and manually tested before
any content generation begins. Do not run the content generation script
until you have confirmed the new UI features work on Module 1 Level 1.

### Rule 5 — Track concept introductions across modules

Keep a running list of concepts that have been introduced.
Pass this list to each new subagent so `new_concepts` is only added
when a concept truly appears for the first time.

Start with an empty set. After each module subagent completes,
update the set with any new concepts it introduced.

### Rule 6 — Report and pause between modules

After each module subagent completes:
1. Report which levels were processed
2. List any new concepts introduced
3. Flag any level where content generation failed or seemed wrong
4. Wait for confirmation before starting the next module

## 1. Project Context

K8sMissions is a terminal-based Kubernetes learning game built in Python with the Rich library. Players fix deliberately broken Kubernetes scenarios using real `kubectl` commands against a local `kind` cluster.

### Key Files

```
k8smissions/
├── engine/
│   ├── engine.py        ← Main game loop and all command handling
│   ├── ui.py            ← All Rich terminal UI (panels, tables, pager)
│   ├── reset.py         ← Level reset: deletes namespace, applies broken.yaml
│   ├── safety.py        ← kubectl safety guards (blocks dangerous commands)
│   ├── certificate.py   ← Module completion certificate generation
│   └── player.py        ← Player name prompt
├── modules/
│   └── module-N-name/
│       └── level-N-name/
│           ├── mission.yaml        ← Level metadata (JSON)
│           ├── broken.yaml         ← Deliberately broken K8s manifests
│           ├── solution.yaml       ← Reference solution
│           ├── validate.sh         ← Bash validation script
│           ├── hint-1.txt          ← First hint
│           ├── hint-2.txt          ← Second hint
│           ├── hint-3.txt          ← Third hint
│           ├── debrief.md          ← Post-level explanation
│           └── common-mistakes.md  ← Common mistakes reference
├── levels.json     ← Pre-built registry (generated by scripts/generate_registry.py)
├── progress.json   ← Player progress (auto-created at runtime)
└── play.sh         ← Game launcher
```

### How the Game Currently Works

1. Player runs `./play.sh` → starts `engine/engine.py`
2. Engine checks cluster health, loads `levels.json`, shows welcome screen
3. For each level: resets namespace (`k8smissions`), applies `broken.yaml`
4. Player investigates with `kubectl` in a second terminal
5. Player types `check` → engine runs `validate.sh` → shows stdout → PASS or FAIL
6. On pass: awards XP, shows debrief.md paginated, advances to next level
7. Player can type `hint` (up to 3), `guide` (shows solution.yaml), `reset`, `skip`, `status`

### Current Problems in V1

1. **`debrief.md` files are placeholders** — contain "Explain X, Y, Z" but never actually explain anything. Example: `## Why It Matters\nExplain exit codes, restartPolicy (Always/OnFailure/Never), back-off delay`
2. **Hints jump to answers** — do not teach the investigation process
3. **New K8s concepts appear without explanation** — "CrashLoopBackOff" is used before being defined
4. **Validation is binary** — PASS or FAIL with no partial progress shown
5. **No guided investigation** — a beginner does not know which kubectl commands to run or how to interpret their output

## 2. V2 Goals

Teach beginners how to **think through** Kubernetes problems, not just copy fixes.

A player who completes Module 1 should be able to:
- Know which kubectl commands to run when a pod is unhealthy
- Interpret `kubectl get pods`, `describe pod`, and `logs` output
- Understand WHY Kubernetes behaves the way it does in each failure scenario
- Know the difference between a lab fix and a production fix

### What V2 Adds

**Four engine features:**
1. Layered validation — show each check passing or failing
2. New Concepts panel — define new K8s terms before each level
3. Investigation Guide — step-by-step investigation walkthrough per level
4. `run` command — execute kubectl from within the game with annotated output

**Content rewrite:**
- All 200 `debrief.md` files with real explanations
- All 600 hint files with investigation-focused content
- `new_concepts` field in mission.yaml for first appearances of concepts
- `investigation.yaml` per level

### What V2 Does NOT Include

- AI coaching at runtime (no API key distribution to players)
- Learning paths or concept graph
- Browser UI
- Multiplayer
- Concept mastery tracking

### 3.1 Layered Validation

#### New validate.sh format

Every `validate.sh` must define and use three helper functions: `_check`, `_pass`, `_fail`.

The engine parses lines that start with `CHECK:`, `PASS:`, or `FAIL:`. Any other lines are ignored by the parser (but still printed raw if no structured output is found).

**Standard template for validate.sh:**

```bash
#!/bin/bash

# Structured output helpers — do not remove
_check() { echo "CHECK:$1:$2"; }
_pass()  { echo "PASS:$1"; }
_fail()  { echo "FAIL:$1:${2:-}"; exit 1; }

# --- Checks ---

_check "resource_exists" "Pod <name> exists in namespace k8smissions"
kubectl get pod <name> -n k8smissions &>/dev/null \
  || _fail "resource_exists" "Pod not found. Did you apply the fix?"
_pass "resource_exists"

_check "pod_running" "Pod phase is Running"
STATUS=$(kubectl get pod <name> -n k8smissions \
  -o jsonpath='{.status.phase}' 2>/dev/null)
[[ "$STATUS" == "Running" ]] \
  || _fail "pod_running" "Current phase: ${STATUS:-Unknown}"
_pass "pod_running"

_check "container_ready" "Container is Ready (accepting traffic)"
READY=$(kubectl get pod <name> -n k8smissions \
  -o jsonpath='{.status.containerStatuses[0].ready}' 2>/dev/null)
[[ "$READY" == "true" ]] \
  || _fail "container_ready" "Container not yet ready"
_pass "container_ready"

# Level-specific checks go here

echo "✅ All checks passed"
exit 0
```

#### Update engine.py — validation display

Replace the current raw-stdout printing in the `check` command handler with a structured parser.

Add this function to `engine.py`:

```python
def display_validation_result(result: subprocess.CompletedProcess) -> None:
    """Parse CHECK/PASS/FAIL lines and display with Rich formatting."""
    from rich import box as rich_box
    from rich.panel import Panel
    from rich.text import Text

    checks: dict[str, dict] = {}
    order: list[str] = []

    for line in result.stdout.splitlines():
        if line.startswith("CHECK:"):
            parts = line.split(":", 2)
            if len(parts) == 3:
                _, id_, label = parts
                checks[id_] = {"label": label, "status": "pending", "message": ""}
                order.append(id_)
        elif line.startswith("PASS:"):
            id_ = line.split(":", 1)[1].strip()
            if id_ in checks:
                checks[id_]["status"] = "pass"
        elif line.startswith("FAIL:"):
            parts = line.split(":", 2)
            id_ = parts[1].strip() if len(parts) > 1 else ""
            message = parts[2].strip() if len(parts) > 2 else ""
            if id_ in checks:
                checks[id_]["status"] = "fail"
                checks[id_]["message"] = message

    # Fall back to raw output if no structured lines found
    if not order:
        if result.stdout:
            console.print(result.stdout.rstrip())
        if result.stderr:
            console.print(f"[yellow]{result.stderr.rstrip()}[/yellow]")
        return

    body = Text()
    for id_ in order:
        c = checks[id_]
        if c["status"] == "pass":
            body.append("  ✓  ", style="bold bright_green")
            body.append(c["label"] + "\n", style="white")
        elif c["status"] == "fail":
            body.append("  ✗  ", style="bold red")
            body.append(c["label"], style="white")
            if c["message"]:
                body.append(f"\n      {c['message']}", style="yellow")
            body.append("\n")
        else:
            body.append("  -  ", style="grey50")
            body.append(c["label"] + "\n", style="grey50")

    passed = result.returncode == 0
    border = "bright_green" if passed else "red"
    title = "[bold bright_green]✓ PASSED[/bold bright_green]" if passed \
        else "[bold red]✗ FAILED[/bold red]"
    console.print(Panel(body, title=title, border_style=border,
                        box=rich_box.ROUNDED, padding=(0, 2)))
```

In `game_loop()`, find every place that prints `result.stdout` after running the validator and replace it with `display_validation_result(result)`. This applies to the `check`, `check-dry`, and `watch` command handlers.

def _show_investigation_guide(path: Path) -> None:
    import yaml as _yaml
    guide = _yaml.safe_load(path.read_text(encoding="utf-8")) or {}
    phases = guide.get("phases", [])
    if not phases:
        return

    parts: list[str] = []

        if phase.get("intro"):
            parts.append(f"{phase['intro'].strip()}\n\n")

def _annotate_pod_output(stdout: str) -> None:
    found = [(status, explanation)
             for status, explanation in _STATUS_ANNOTATIONS.items()
             if status in stdout]
    if not found:
        return
    from rich.text import Text
    t = Text("\n")
    for status, explanation in found:
        t.append(f"  {status}", style="bold yellow")
        t.append(f"  →  {explanation}\n", style="grey70")

Every `debrief.md` generated must pass this checklist. If a generated file fails any item, regenerate or edit manually:

**Example of a passing debrief for level-2-crashloop:**

```markdown
# The Infinite Restart

## What Happened

The crashloop-pod was configured with an `args` field that referenced
`/scripts/start.sh` — a script that does not exist inside the container
image. When Kubernetes launched the container, the shell immediately
exited with code 127 (command not found in Linux).

Kubernetes saw the non-zero exit code and treated it as a failure. It
restarted the container automatically. The same thing happened again.
And again. This is the loop that produces the CrashLoopBackOff status.

To prevent this loop from consuming excessive resources, Kubernetes adds
an increasing delay between restarts: 10s, then 20s, 40s, 80s, up to a
maximum of 5 minutes. That increasing delay is the "BackOff" in
CrashLoopBackOff.

## The Evidence That Revealed the Cause

```bash
# The status showed a loop, not a one-time crash
kubectl get pod crashloop-pod -n k8smissions
# NAME            READY   STATUS             RESTARTS   AGE
# crashloop-pod   0/1     CrashLoopBackOff   5          4m

# The previous container log showed the exact error
kubectl logs crashloop-pod -n k8smissions --previous
# /bin/sh: /scripts/start.sh: not found

# The spec showed where the path was configured
kubectl get pod crashloop-pod -n k8smissions -o yaml | grep -A3 args
# args:
# - /scripts/start.sh
```

## Why Kubernetes Behaved This Way

Kubernetes does not distinguish between a crash caused by a bug and a
crash caused by a misconfiguration. Both result in a non-zero exit code,
and both trigger a restart. The kubelet on the node is responsible for
running containers and restarting them according to the pod's
`restartPolicy` (default: Always).

The exponential backoff exists to prevent a broken container from
continuously consuming CPU and generating log noise. Without backoff,
a misconfigured container could crash thousands of times per hour.

## The Fix

Removing the `args` field allowed the container to use its image's
default ENTRYPOINT, which is the nginx web server. The container started
successfully because the nginx binary is part of the image.

Alternatively, the `args` value could have been corrected to reference a
script that actually exists inside the image.

## Lab vs Production

**In this lab:** The pod was recreated without the broken args field.
Directly deleting and recreating a standalone pod is acceptable in a
local learning environment.

**In production:** This pod would be managed by a Deployment. You would
never delete or edit a pod directly. Instead, you would update the
Deployment's pod template (`spec.template.spec.containers[0].args`),
and the Deployment controller would perform a rolling update —
creating new pods with the fix before removing the broken ones. Editing
a pod directly in production means your fix disappears the next time
the Deployment recreates the pod.

## Commands to Remember

| Command | When to use it |
|---------|---------------|
| `kubectl logs <pod> -n <ns> --previous` | Read logs from a container that has already crashed |
| `kubectl get pod <pod> -o yaml` | See the full pod spec including args and command |
| `kubectl describe pod <pod>` | See events, state history, and last exit code |

## One Thing to Remember

When a container crashes, always read `--previous` logs — the current
container may have just started and have nothing to show yet.
```
