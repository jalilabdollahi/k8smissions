OVERVIEW
────────
This is the entry point for all players. World 1 covers everything
a Kubernetes beginner needs to be productive: running Pods, reading
logs, understanding scheduling, Services, ConfigMaps, and Secrets.

Every level is intentionally broken. Your job is to find the bug
and fix it. The cluster namespace is "k8smissions".

Difficulty   : Beginner → Early Intermediate
Total levels : 15
Total XP     : 2,050
Cluster type : kind (local)

SKILLS COVERED
──────────────
- kubectl get / describe / logs / apply / delete
- Pod lifecycle (Pending → Running → CrashLoopBackOff → OOMKilled)
- Namespaces
- Labels and selectors
- Services (ClusterIP, targetPort)
- Environment variables, ConfigMaps, Secrets
- Multi-container Pods (sidecar pattern)
- Init containers
- Resource requests and limits
- nodeSelector
- Log-driven debugging

LEVEL 1 — first-pod
  Name        : "Hello, Kubernetes!"
  Difficulty  : beginner
  XP          : 100
  Time est.   : 5m
  Folder      : worlds/world-1-foundations/level-1-first-pod/
  Description : A Pod was created but it is stuck in Error state.
                The user defined a bad entrypoint command.
  Broken state: nginx Pod with command: ["badcommand"]
  Fix         : Remove the command override so nginx starts normally
  Validate    : kubectl get pod hello-pod -n k8smissions → Running
  Concepts    : pod, command, entrypoint, CrashLoopBackOff
  Hint 1      : Check what state the pod is in
  Hint 2      : kubectl describe pod hello-pod -n k8smissions
  Hint 3      : The command field overrides the container's entrypoint
  Debrief     : Explain ENTRYPOINT vs CMD in Docker and how Kubernetes
                command/args maps to them

LEVEL 2 — crashloop
  Name        : "The Infinite Restart"
  Difficulty  : beginner
  XP          : 100
  Time est.   : 8m
  Folder      : worlds/world-1-foundations/level-2-crashloop/
  Description : Pod is in CrashLoopBackOff. The container exits immediately
                because the args point to a script that does not exist.
  Broken state: busybox Pod with args: ["/scripts/start.sh"]
                (the file does not exist in the image)
  Fix         : Change args to ["/bin/sh", "-c", "sleep 3600"]
  Validate    : Pod Running for > 30s, restartCount = 0
  Concepts    : CrashLoopBackOff, args, restartPolicy, exit codes
  Hint 1      : Look at the RESTARTS column
  Hint 2      : kubectl logs crashloop-pod -n k8smissions (previous)
  Hint 3      : The script referenced in args does not exist in busybox
  Debrief     : Explain exit codes, restartPolicy (Always/OnFailure/Never),
                back-off delay

LEVEL 4 — pending-resources
  Name        : "Too Greedy"
  Difficulty  : beginner
  XP          : 150
  Time est.   : 10m
  Folder      : worlds/world-1-foundations/level-4-pending-resources/
  Description : Pod stuck in Pending. It requests 100 CPUs — far more
                than any node has available.
  Broken state: Pod with resources.requests.cpu: "100"
  Fix         : Set cpu request to "100m" (100 millicores)
  Validate    : Pod scheduled and Running
  Concepts    : resource requests, resource limits, scheduling, millicores
  Hint 1      : kubectl describe pod pending-pod -n k8smissions → Events
  Hint 2      : "0/1 nodes available: Insufficient cpu"
  Hint 3      : 1 CPU = 1000m. "100" means 100 full CPUs.
  Debrief     : Explain requests vs limits, how scheduler uses requests,
                QoS classes (Guaranteed / Burstable / BestEffort)

LEVEL 6 — namespace-mismatch
  Name        : "Lost in Space"
  Difficulty  : beginner
  XP          : 150
  Time est.   : 10m
  Folder      : worlds/world-1-foundations/level-6-namespace-mismatch/
  Description : A Pod and its Service were deployed to the "default" namespace
                but the app expects to be in "k8smissions". Service lookup fails.
  Broken state: Pod + Service in namespace: default
  Fix         : Delete and recreate both in namespace: k8smissions
  Validate    : Pod and Service exist in k8smissions namespace and are reachable
  Concepts    : namespaces, resource isolation, cross-namespace DNS
  Hint 1      : kubectl get pods --all-namespaces
  Hint 2      : Nothing in k8smissions namespace?
  Hint 3      : Resources exist in "default" — wrong namespace
  Debrief     : Explain namespace isolation, RBAC per namespace, default namespace

LEVEL 7 — labels-selectors

  XP          : 150

  Folder      : worlds/world-1-foundations/level-7-labels-selectors/
  Description : Service exists but has zero endpoints. Pod labels don't match
                the service selector.
  Broken state: Pod has label app: backend-v1

  Fix         : Either update pod label to "backend" or service selector to "backend-v1"

  Concepts    : labels, selectors, service endpoints, kubectl get ep
  Hint 1      : kubectl get endpoints labels-svc -n k8smissions
  Hint 2      : kubectl describe svc labels-svc -n k8smissions → Selector
  Hint 3      : Compare pod labels to service selector — there is a mismatch
  Debrief     : Explain how Services discover pods via label selectors,
                importance of consistent labeling strategy

  Debrief     : ConfigMap as env variables vs mounted files, immutable ConfigMaps

LEVEL 10 — secrets-basics
  Name        : "Forbidden Credentials"
  Difficulty  : beginner
  XP          : 150
  Time est.   : 10m
  Folder      : worlds/world-1-foundations/level-10-secrets-basics/
  Description : Pod references Secret "db-credentials" that doesn't exist.
                Pod fails with CreateContainerConfigError.
  Broken state: Pod references secret db-credentials (not created)
  Fix         : Create Secret with username and password (base64 encoded)
  Validate    : Pod Running, secret values accessible as env vars
  Concepts    : Secret, base64 encoding, envFrom, secretKeyRef
  Hint 1      : kubectl describe pod secret-pod -n k8smissions
  Hint 2      : "secret 'db-credentials' not found"
  Hint 3      : echo -n 'mypassword' | base64  → use in secret data field
  Debrief     : Secrets vs ConfigMaps, base64 is NOT encryption,
                external secret stores (Vault, Sealed Secrets)

LEVEL 15 — node-selector
  Name        : "Won't Schedule"
  Difficulty  : intermediate
  XP          : 200
  Time est.   : 12m
  Folder      : worlds/world-1-foundations/level-15-node-selector/

                disktype=ssd, but no node has that label.
  Broken state: nodeSelector: disktype: ssd (no node has this label)
  Fix         : Either remove the nodeSelector or add the label to a node:

  Validate    : Pod scheduled and Running
  Concepts    : nodeSelector, node labels, scheduling constraints,
                affinity vs nodeSelector
  Hint 1      : kubectl describe pod nodeselector-pod -n k8smissions
  Hint 2      : "0/1 nodes available: node(s) didn't match nodeSelector"
  Hint 3      : Check node labels: kubectl get nodes --show-labels
  Debrief     : nodeSelector vs nodeAffinity, taints vs labels,
                when to constrain scheduling
