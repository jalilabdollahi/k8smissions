OVERVIEW
────────
Module 6 covers
the observability pillar of Kubernetes operations: metrics collection,
log analysis, event interpretation, resource monitoring, probe tuning,
and debugging live production issues without restarting anything.

Real-world SRE skills are the focus here.

Difficulty   : Advanced
Total levels : 10
Total XP     : 2,900
Cluster type : kind (local)
New concepts : metrics-server, kubectl top, events API, exec debugging,
               Prometheus basics, alert patterns

SKILLS COVERED
──────────────
- kubectl top nodes / top pods (resource usage)
- kubectl describe events
- kubectl exec for live debugging
- metrics-server installation and health
- Liveness vs readiness vs startup probes (correct usage)
- OOMKilled diagnosis and memory tuning
- Node pressure conditions (MemoryPressure, DiskPressure, PIDPressure)
- Container restart cause analysis
- Probe initialDelaySeconds / timeoutSeconds tuning
- kubectl get events --sort-by='.lastTimestamp'
- Debugging without restarting (ephemeral containers)
- Prometheus scraping annotations intro

LEVELS
───────────────────────────────────────────────────────────────────

LEVEL 1 — metrics-server-missing  (level-1-metrics-server)
  Name        : "Flying Blind"
  Difficulty  : intermediate
  XP          : 200
  Description : Team complains "kubectl top pods" returns error:
                "metrics not available yet". metrics-server not installed.
                HPA also broken because of this.
  Broken state: No metrics-server component in cluster
  Fix         : Apply metrics-server manifest (bundled in level folder)
                Wait for it to become Ready, verify kubectl top pods works
  Validate    : kubectl top pods -n k8smissions returns CPU/Memory values
  Concepts    : Metrics API, metrics-server, Resource Metrics Pipeline,
                Custom Metrics API, Prometheus Adapter

LEVEL 2 — pod-events-analysis  (level-2-pod-events)
  Name        : "Read the Events"
  Difficulty  : beginner
  XP          : 150
  Description : Pod is failing but logs are empty. The issue is an
                image pull problem only visible in cluster events.
  Broken state: Pod with wrong image tag (private registry, auth error)
  Fix         : kubectl get events -n k8smissions --sort-by='.lastTimestamp'
                Find the ImagePullBackOff event → fix image reference
  Validate    : Pod Running
  Concepts    : kubectl events, Event objects, reason/message fields,
                Normal vs Warning events, event TTL (1 hour default)

LEVEL 3 — resource-top  (level-3-resource-top)
  Name        : "Who's Eating All the CPU?"
  Difficulty  : intermediate
  XP          : 250
  Description : Node is under CPU pressure. Multiple pods running.
                Find the CPU-hungry pod and scale it down or limit it.
  Broken state: One pod has no CPU limit and is consuming 90% of CPU
  Fix         : Use kubectl top pods to identify the culprit
                Add resources.limits.cpu: "200m" to the greedy pod
  Validate    : Node CPU back under 70%, other pods healthy
  Concepts    : kubectl top, resource limits vs requests, noisy-neighbor,
                LimitRange to enforce defaults

LEVEL 4 — log-aggregation  (level-4-log-aggregation)
  Name        : "Logs Across Restarts"
  Difficulty  : intermediate
  XP          : 300
  Description : App crashed and restarted 5 times. Need to see the logs
                from the PREVIOUS crash, not current instance.
  Broken state: Pod restarting; current logs show startup only
  Fix         : kubectl logs <pod> --previous -n k8smissions
                Identify the crash message (OOM / missing config / etc)
  Validate    : Player outputs the crash reason from previous logs
  Concepts    : kubectl logs --previous, --since, --tail, multi-container logs,
                log rotation, log aggregation tools (Loki, EFK, CloudWatch)

LEVEL 5 — liveness-vs-readiness  (level-5-probe-types)
  Name        : "Wrong Probe, Wrong Job"
  Difficulty  : intermediate
  XP          : 250
  Description : A liveness probe is checking /api/v1/ready (readiness logic).
                When the app is warming up, liveness kills and restarts it —
                infinite loop. The fix is to use readiness for warmup checks.
  Broken state: livenessProbe.httpGet.path: /api/v1/ready (slow endpoint)
  Fix         : Move the slow check to readinessProbe,
                use a lightweight /healthz for livenessProbe
  Concepts    : Liveness vs readiness vs startup probe responsibilities,
                probe endpoints, startupProbe to delay liveness checks

LEVEL 6 — container-restart-cause  (level-6-restart-cause)
  Name        : "Why Does It Keep Restarting?"
  Difficulty  : intermediate
  XP          : 300
  Description : Pod shows RESTARTS: 8 over the last hour. No obvious error.
                Must use events, logs, and describe to find root cause.
                Root cause: OOMKilled (not CrashLoopBackOff from bad code).
  Broken state: Pod memory limit 64Mi, app uses 128Mi under load
  Fix         : kubectl describe pod → find "OOMKilled" in Last State
                Increase memory limit to 256Mi
  Concepts    : Exit codes (137=SIGKILL/OOM, 1=crash, 0=clean),
                kubectl describe lastState, memory profiling

LEVEL 7 — node-pressure  (level-7-node-pressure)
  Name        : "Node Under Pressure"
  Difficulty  : advanced
  XP          : 350
  Description : Pods randomly evicted from a node. Node shows
                MemoryPressure condition = True.
  Broken state: Node with MemoryPressure condition (simulated via many pods)
  Fix         : kubectl describe node → identify pressure condition
                Identify and remove / limit high-memory pods
                OR cordon node and reschedule workloads
  Concepts    : Node conditions (MemoryPressure, DiskPressure, PIDPressure),
                eviction threshold, cordon/drain, node lifecycle

LEVEL 8 — probe-tuning  (level-8-probe-tuning)
  Name        : "App Takes 60s to Start"
  Difficulty  : intermediate
  XP          : 300
  Description : A Java Spring Boot app takes ~60 seconds to start.
                Liveness probe fires at 10s, kills the pod, restart loop.
  Broken state: livenessProbe.initialDelaySeconds: 5 (too short)
  Fix         : Option A: Increase initialDelaySeconds: 90
                Option B (better): Add startupProbe with
                  failureThreshold: 30, periodSeconds: 5 (= 150s max wait)
  Concepts    : startupProbe, initialDelaySeconds, failureThreshold,
                periodSeconds, probe math, slow-start applications

LEVEL 9 — exec-debugging  (level-9-exec-debugging)
  Name        : "Diagnose the Live Pod"
  Difficulty  : advanced
  XP          : 300
  Description : Pod is Running but app returns 500 errors. No logs.
                Must exec into container and inspect the filesystem/config.
  Broken state: App config file has wrong DB host IP (hardcoded)
  Fix         : kubectl exec -it <pod> -- /bin/sh
                cat /app/config.json → find wrong DB_HOST value
                Patch the ConfigMap and restart pod
  Concepts    : kubectl exec, ephemeral containers (kubectl debug),
                live debugging, configuration inspection

LEVEL 10 — cluster-events  (level-10-cluster-events)
  Name        : "The Timeline of Failure"
  Difficulty  : advanced
  XP          : 500
  Description : A 15-minute production incident happened. Multiple events
                logged across namespaces. Player must reconstruct the incident
                timeline from events and logs + write a brief post-mortem.
  Broken state: Cluster with pre-seeded events from a simulated incident:
                OOMKill → pod eviction → node pressure → cascading failures
  Fix         : kubectl get events -A --sort-by='.lastTimestamp' | tee timeline.txt
                Identify root cause event and chain
  Concepts    : Event timeline, incident post-mortem, MTTD (Mean Time to Detect),
                observability → actionability, SRE principles
